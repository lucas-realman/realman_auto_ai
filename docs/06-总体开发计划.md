## 1. 项目概况与目标

### 1.1 项目定位

| 项 | 内容 |
|----|------|
| **项目名称** | 睿尔曼智能 CRM（Sirus AI CRM） |
| **核心理念** | Agent 就是 CRM 本身 —— 用户通过对话完成所有 CRM 操作 |
| **架构模式** | AI-Native：5 层架构（接入 → NLU → Agent → Tool → 数据） |
| **开发模式** | 全自动化流水线 — Orchestrator 调度 + aider 编码 + LLM 自动 Review（详见 04-§13、08-自动化开发流水线） |
| **测试模式** | 开发即测试 —— 代码推送即触发自动评估（详见 05-测试方案） |

### 1.2 一期目标

> **一句话**：Agent 能通过钉钉完成基础 CRM 操作 + 关键 AI 洞察。

| # | 目标 | 衡量标准 |
|---|------|---------|
| G1 | 自然语言 CRM 操作 | 销售在钉钉说"帮我创建线索/客户/商机"，Agent 完成操作 |
| G2 | AI 智能洞察 | 线索自动评分、商机赢率预测、客户价值分析 |
| G3 | Web 管理看板 | 仪表盘 + 线索/客户/商机/活动列表 + 详情页 |
| G4 | 钉钉深度集成 | 机器人对话、互动卡片、SSO 单点登录 |
| G5 | 系统可靠性 | vLLM 降级保护、数据备份、操作审计 |

### 1.3 开发团队与资源

| 角色 | 运行位置 | 职责 |
|------|---------|------|
| **方案设计者 (你)** | 本地 | 编写/更新设计文档 (`docs/`) 和接口契约 (`contracts/`)，查看钉钉报告，异常时决策 |
| **Orchestrator** | orchestrator (专用服务器) | 解析设计文档 → 生成编码任务 → SSH 分发 → 收集结果 → 自动 Review → 判定重试/通过（详见 08-§2） |

### 2.3 关键路径上的瓶颈

| # | 瓶颈 | 风险 | 缓解措施 |
|---|------|------|---------|
| 1 | **Prompt 调优** | Agent 效果反复迭代，无法量化完成标准 | 云端多模型 Judge 自动评估（05-§4），设定 ≥4.0/5.0 分通过线 |
| 2 | **LLM Review 覆盖率** | 自动 Review 可能遗漏复杂业务逻辑 | 三层自动 Review（静态+契约+设计符合度），Sprint 末人工抽检（08-§4） |
| 3 | **钉钉集成调试** | 钉钉回调、SSO 等需要外部服务联调 | S3-4 优先打通钉钉回调最小闭环（S1-2 已注册应用） |
| 4 | **端到端联调** | 5 台机器跨网络调用，故障定位慢 | 接口契约先行（04-§13.4），减少对齐问题 |

### 2.4 每 Sprint 的节奏

```
Sprint 标准节奏 (7-10 天, Orchestrator 7×24 含周末)：

Day 1       Sprint 启动 (Orchestrator 自动)
            ├── 读取 07-Sprint任务卡 中本 Sprint 的任务
            ├── 自动解析 → 生成编码任务队列
            ├── 5 台机器 git pull 同步
            ├── 钉钉通知: "Sprint X 启动，共 XX 个任务"
            └── 你: 确认设计文档无需修改（通常无需操作）

Day 2~N-2   全自动并行开发 (Orchestrator 24h 不间断，含周末)
            ├── 4 台机器: aider 全天轮转编码 → 自动 commit
            ├── Orchestrator: 三层自动 Review → 通过/重试
            ├── orchestrator: 代码推送即本地触发 pytest + 云端API多模型 Judge（零延迟）
            ├── 钉钉: 每日 18:00 自动发送开发摘要
            └── 你: 查看钉钉报告，有升级任务时再介入

Day N-1     自动联调 + 修复
            ├── Orchestrator 触发端到端集成测试
            ├── 失败项自动生成修复任务 → aider 修复
            └── 性能基线自动检查

Day N       Sprint Review (半自动)
            ├── Orchestrator 生成 Sprint 完成报告 → 钉钉
            ├── 自动验证里程碑通过标准
            ├── 你: 确认是否进入下一 Sprint
            └── Orchestrator 自动打 tag + 准备下一 Sprint

注: N = Sprint 天数 (S3-4: 10d, S5-6: 10d, S7-8: 8d)
    S1-2 已完成，以上节奏从 S3-4 起适用
```

## 3. 风险与应对策略

### 3.1 风险登记册

| # | 风险 | 概率 | 影响 | 等级 | 应对策略 | 负责 |
|---|------|------|------|------|---------|------|
| R1 | vLLM 推理性能不达标（TTFT>3s） | 低 | 高 | 🟡 | ✅ S1-2 已验证基本可用；如后续劣化: 调整 tp_size/batch_size，Qwen3.5-35B-A3B 备选 | 持续监控 |
| R2 | Agent Prompt 效果不稳定 | 高 | 高 | 🔴 | 云端多模型 Judge 自动评估 + Prompt A/B 测试框架；设定评分基线 ≥4.0 | 持续 |
| R3 | 钉钉应用审核延迟 | 低 | 高 | 🟡 | Sprint 1 第 1 天就提交申请；先用测试模式开发 | Sprint 1 |
| R4 | 云端 API 调用超时/限流 | 中 | 中 | 🟡 | 本地降级路径；异步重试队列；云端调用脱敏（04-§7.8） | Sprint 3+ |
| R5 | Orchestrator 调度引擎故障 | 低 | 高 | 🟡 | systemd 自动重启 + 看门狗脚本（08-§6.4）；状态持久化到 Redis | Sprint 1 |
| R6 | aliyun-agent 网络不稳定 | 低 | 低 | ⭐ | 测试已迁至 orchestrator 本地执行，不再依赖云端网络；aliyun-agent 仅用于 Staging 预发布验收 | 持续 |
| R7 | 数据库迁移冲突 | 低 | 中 | 🟢 | Alembic 由 mac_min_8T 独占执行，Orchestrator 串行调度 DB 迁移任务 | 持续 |
| R8 | MoE 模型显存溢出 | 低 | 高 | 🟡 | 监控 GPU 显存；并发限制；设置 max_model_len 上限 | Sprint 1 |

### 3.3 应急预案

| 场景 | 触发条件 | 操作 |
|------|---------|------|

| **Sprint 延期** | 里程碑未达标 | 砍掉非关键功能（云端 Agent、高级卡片），聚焦核心 CRUD + 对话链路 |
