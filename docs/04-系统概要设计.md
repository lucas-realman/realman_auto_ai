## 2. 核心理念：AI-Native CRM

### 2.1 传统 CRM vs AI-Native CRM

| 维度 | 传统 CRM | AI-Native CRM (本方案) |
|------|---------|----------------------|
| **交互方式** | 人操作表单/菜单 | 人与 Agent 对话，Agent 执行一切 |
| **数据录入** | 人手动填写字段 | Agent 从对话/邮件/通话中自动提取 |
| **业务流程** | 硬编码工作流 (if-else) | Agent 基于上下文动态决策 |
| **规则引擎** | 预定义规则触发 | Agent 理解业务语义，自主判断 |
| **报表分析** | 预设报表模板 | Agent 实时分析，自然语言问答 |
| **通知提醒** | 基于规则的定时触发 | Agent 感知业务变化，主动推送 |
| **客户画像** | 标签体系 + 手动标注 | Agent 持续理解客户，动态生成 |
| **销售辅导** | 经理人工指导 | Agent 分析行为模式，个性化建议 |
| **系统集成** | API 对接 + ETL | Agent 理解跨系统语义，智能编排 |
| **用户界面** | 复杂的 Web 应用 | 钉钉对话(主) + 轻量看板(辅) |

### 2.2 一句话定位

> **Sirus 不是一个带 AI 功能的 CRM，而是一个理解销售业务的 AI Agent，它的记忆就是 CRM 数据。**

### 2.3 设计原则

| 原则 | 说明 |
|------|------|
| **Agent-First** | 每个 CRM 功能首先设计为 Agent 能力，而非表单界面 |
| **对话即操作** | 用户通过自然语言完成所有 CRM 操作，零学习成本 |
| **感知即录入** | Agent 从交互中自动感知并记录信息，而非等人填写 |
| **理解即流程** | Agent 理解业务语义后自主推进流程，无需硬编码工作流 |
| **记忆即数据** | 结构化数据库是 Agent 的长期记忆，Agent 自主读写 |
| **主动即服务** | Agent 不需要等待指令，主动发现问题、推送洞察、建议行动 |
| **可降级** | AI 不可用时，基础 CRUD 仍通过 Web 看板可操作 |

### 3.2 与传统架构的关键区别

```
传统 CRM 架构:
  用户 → Web UI → 业务服务层 → 数据库
                     ↕
                  AI 辅助层 (可有可无)

AI-Native 架构 (本方案):
  用户 → 钉钉对话 → Agent 大脑 → 工具调用 → 记忆读写
                       ↕              ↕
                    事件总线      外部系统
                       ↕
                   主动推送 → 用户
```

**核心变化**：没有「业务服务层」——Agent 大脑就是业务逻辑。传统 CRM 里用代码写的 if-else 流程，在这里由 Agent 理解语义后动态决策。

## 4. CRM 功能的 AI 化实现

> 传统 CRM 的每个模块，不再是「界面 + 后端逻辑 + 数据库」，而是「Agent 能力 + 工具 + 记忆」。

### 4.1 总览：CRM 模块 → Agent 能力映射

| 传统 CRM 模块 | AI-Native 实现方式 | 触发方式 |
|--------------|-------------------|---------|
| **线索管理** | Agent 自动采集/评分/分配/跟进编排 | 事件驱动 + 对话 |
| **客户管理** | Agent 持续构建客户认知，动态维护画像 | 事件驱动 + 定时 |
| **联系人管理** | Agent 自动识别决策链，维护关系图谱 | 数据变更事件 |
| **商机管理** | Agent 全程跟踪/预测赢率/推荐行动/检测卡点 | 事件驱动 + 对话 |
| **产品管理** | Agent 管理产品知识库，智能推荐产品组合 | 对话 + 知识检索 |
| **合同管理** | Agent 审查条款/预检/到期追踪 | 事件驱动 + 定时 |
| **订单管理** | Agent 跟踪履约/异常检测/进度播报 | 外部系统事件 |
| **回款管理** | Agent 追踪回款/预警逾期/推荐催收策略 | 定时 + 阈值事件 |
| **目标管理** | Agent 智能拆解/实时追踪/差距分析/行动建议 | 定时 + 对话 |
| **行为管理** | Agent 自动记录/分析模式/生成日报周报 | 事件驱动 + 定时 |
| **数据分析** | Agent 自然语言问答/主动推送洞察/预测 | 对话 + 定时 |
| **钉钉集成** | Agent 通过钉钉与人交互（这就是界面） | 全时在线 |
| **审批流程** | Agent 在钉钉内完成全流程审批 | 事件驱动 |
| **系统集成** | Agent 理解跨系统语义，智能编排调用 | 事件驱动 |

### 4.2 示例：传统流程 vs AI-Native 流程

#### 示例一：新建线索

```
传统 CRM:
  销售 → 打开CRM → 点击"新建线索" → 填写15个字段 → 保存
  (耗时10分钟，可能漏填)

AI-Native:
  销售在钉钉说: "刚在展会认识了一个做教育的客户，张总，
                 电话13800138000，他们想采购一批机械臂用于教学"
  Agent:
    1. 理解意图 → 创建线索
    2. 提取实体 → 姓名:张总, 电话:13800138000, 行业:教育, 需求:机械臂教学
    3. 调用天眼查工具 → 自动补全企业信息
    4. 调用线索评分工具 → 计算评分 78分 (教育行业+明确需求)
    5. 调用线索CRUD工具 → 写入数据库
    6. 调用钉钉工具 → 推送确认卡片给销售
    7. 产生事件 → 触发分配Agent分析最优销售
  (耗时15秒，信息完整)
```

#### 示例二：商机阶段推进

```
传统 CRM:
  销售 → 打开CRM → 找到商机 → 手动改阶段 → 填写推进记录 → 保存
  经理 → 每周打开漏斗报表 → 人工分析哪些商机卡住了

AI-Native:
  销售在钉钉说: "今天给XX客户做了产品演示，反馈很好，他们下周出方案评审"
  Agent:
    1. 理解：商机推进 → 产品演示完成 → 客户正面反馈 → 即将方案评审
    2. 自动更新商机阶段: 需求确认 → 方案评审
    3. 自动记录拜访活动 + AI摘要
    4. 重新计算赢率: 45% → 62% (正面反馈 + 推进顺利)
    5. 推荐下一步: "建议准备技术方案，安排售前工程师下周一前完成"
    6. 推送给经理: "XX商机推进到方案评审阶段，赢率提升至62%"

  与此同时 (主动型Agent):
    巡检Agent每日扫描: "YY商机在需求确认阶段已停留22天(平均12天)，
    建议联系客户确认是否需要调整方案" → 推送给负责销售
```

#### 示例三：回款管理

```
传统 CRM:
  财务 → 手动核对银行流水 → 手动标记回款 → 手动通知销售

AI-Native:
  Agent(主动型):
    1. 每日检查应收账款计划
    2. 发现某笔50万即将到期(3天后)
    3. 查询客户历史回款习惯: 平均延迟5天
    4. 查询客户最新状态: 健康度评分降了10分
    5. 综合判断: 中等逾期风险
    6. 推送给客户经理: "XX客户50万回款3天后到期，历史有延迟倾向，
       建议今天电话提醒。参考话术：..."
    7. 如果逾期7天，自动升级: 推送给销售经理 + 建议催收策略
```

### 4.3 CRM 功能的三种 AI 化模式

| 模式 | 说明 | 适用场景 | 举例 |
|------|------|---------|------|
| **对话即操作** | 用户说一句话，Agent 完成完整操作 | 所有 CRUD 操作 | "帮我创建一个线索" "把这个商机推到下一阶段" |
| **感知即记录** | Agent 从对话/事件中自动提取信息并记录 | 拜访记录、跟进记录、客户信息更新 | 销售聊天内容中提到客户新需求 → 自动更新客户画像 |
| **主动即服务** | Agent 不等指令，基于事件/时间主动行动 | 预警、评分、规划、报告 | 晨间推送、线索衰减预警、季度复盘 |

## 5. Agent 架构详细设计

### 5.1 Supervisor Agent (总调度)

Supervisor 是 Agent 大脑的中枢，负责：

```
用户消息 / 系统事件
        │
        ▼
┌───────────────────────────────────────┐
│         Supervisor Agent              │
│                                       │
│  ① 意图理解                           │
│     "用户想干什么？这个事件意味着什么？"  │
│                                       │
│  ② 上下文加载                          │
│     从记忆层加载相关信息                 │
│     (当前对话历史、用户角色、相关业务数据) │
│                                       │
│  ③ 任务规划                            │
│     "需要做哪些步骤？调用哪些工具？"      │
│     "需要委派给哪个专业 Agent？"         │
│                                       │
│  ④ 执行编排                            │
│     按计划顺序/并行执行工具调用           │
│     处理中间结果、判断是否需要追加步骤     │
│                                       │
│  ⑤ 风险评估                            │
│     "这个操作是否需要人确认？"            │
│     低风险→自动执行 / 高风险→请求确认     │
│                                       │
│  ⑥ 结果编排                            │
│     将执行结果组织为自然语言回复           │
│     决定推送渠道和形式                   │
│                                       │
└───────────────────────────────────────┘
```

### 5.2 Agent 类型与职责

#### 响应型 Agent (对话触发)

| Agent | 能力 | 可用工具 |
|-------|------|---------|
| **销售助手 Agent** | 理解销售的自然语言指令，执行CRM操作 | 全部CRM CRUD工具、钉钉工具 |
| **数据问答 Agent** | 理解管理层的数据问题，查询分析并回答 | 报表查询工具、统计聚合工具、趋势分析工具 |
| **合同审查 Agent** | 理解合同条款，对比模板，标记风险 | 合同模板工具、风险评估工具 |

#### 主动型 Agent (事件/定时触发)

| Agent | 能力 | 触发条件 |
|-------|------|---------|
| **巡检 Agent** | 扫描全部业务数据，发现异常和遗漏 | 每日定时 |
| **预警 Agent** | 监控阈值突破，推送分级预警 | 数据变更事件 + 定时检查 |
| **规划 Agent** | 为每位销售编排工作计划 | 每日晨间 + 周一 |
| **催收 Agent** | 追踪应收账款，分级催收策略 | 每日定时 + 逾期事件 |
| **晨报 Agent** | 生成个性化每日简报 | 每日 08:00 |

#### 分析型 Agent (数据驱动)

| Agent | 能力 | 输出 |
|-------|------|------|
| **客户洞察 Agent** | 构建/更新客户画像，评估健康度 | 客户画像JSON + 风险提示 |
| **商机预测 Agent** | 预测赢率，分析卡点，推荐行动 | 赢率分数 + 推进建议 |
| **漏斗分析 Agent** | 分析销售漏斗健康度，预测收入 | 漏斗报告 + 达标预测 |
| **行为分析 Agent** | 分析销售行为模式，提炼最佳实践 | 行为报告 + 辅导建议 |
| **线索评分 Agent** | 多维度评估线索质量 | 0-100分 + 评分依据 |

### 5.3 Agent 协作模式

```
场景：新线索进入系统

外部事件: 展会扫码表单提交
    │
    ▼
Supervisor Agent
    │
    ├─▶ 线索评分 Agent
    │     └─ 使用: 评分计算工具、天眼查工具
    │     └─ 输出: 评分 82, 高质量
    │
    ├─▶ 客户洞察 Agent
    │     └─ 使用: 天眼查工具、行业知识检索
    │     └─ 输出: 企业画像 + 潜在需求分析
    │
    └─▶ 规划 Agent (基于评分结果)
          └─ 使用: 数据查询工具(销售负载)、匹配算法
          └─ 输出: 推荐分配给销售A(行业匹配+低负载)
    │

## 7. 技术选型

> AI-Native 架构下，技术选型围绕「Agent 运行环境」组织，而非传统的前后端分层。

### 7.1 Agent 运行时 (核心)

| 组件 | 选型 | AI-Native 角色 |
|------|------|---------------|
| LLM 推理引擎 | **vLLM** | Agent 的「思考能力」，高性能推理，PagedAttention，OpenAI 兼容 API |
| LLM 模型 | **Qwen3-30B-A3B** (一期) | MoE 架构，激活 3B 参数，推理快，2×4090 可部署；后续可升级 Qwen3.5-35B-A3B |
| Agent 框架 | **自研轻量框架** | 基于 vLLM Tool-Calling 原生能力 + Prompt 模板引擎，不引入 LangChain 等重型框架 |
| LLM 客户端 | **OpenAI SDK** (Python) | vLLM 兼容 OpenAI API，标准化接口，模型可切换 |
| Prompt 管理 | **Jinja2 模板 + Git 版本控制** | Prompt 即代码，支持 A/B 测试和版本回滚 |
| 向量数据库 | **pgvector** (PostgreSQL 扩展) | Agent 的语义记忆，不额外引入独立向量数据库，降低运维复杂度 |
| Embedding 模型 | **一期：阿里云百炼 Embedding API** (text-embedding-v3)<br>二期：本地 **BGE-M3**（4090 与 vLLM A 并存） | 中文语义理解，用于知识库检索和经验匹配。一期知识量小走云端 API（零部署、成本~几元/月），二期知识量上来后迁本地 |

### 7.2 记忆层 (数据存储)

| 组件 | 选型 | AI-Native 角色 |
|------|------|---------------|
| 结构化记忆 | **PostgreSQL 16** + pgvector | Agent 的长期记忆：CRM 业务数据 + 向量索引（一库两用） |
| 工作记忆 | **Redis 7** | Agent 的短期记忆：对话上下文、待确认队列、推理链缓存 |
| 事件总线 | **Redis Stream** | Agent 的神经网络：业务事件发布/订阅，驱动主动型 Agent |
| ORM | **SQLAlchemy 2.0** (async) | 工具层访问结构化记忆的接口 |
| 数据库迁移 | **Alembic** | 记忆结构的版本管理 |

### 7.3 服务层 (Agent 的身体)

| 组件 | 选型 | AI-Native 角色 |
|------|------|---------------|
| Web 框架 | **FastAPI** (Python) | Agent 与外界的通讯接口（接收钉钉回调、暴露 API、SSE 流式输出） |

### 7.7 关键技术决策说明

| 决策 | 选择 | 理由 |
|------|------|------|
| Agent 框架自研 vs LangChain | **自研** | LangChain 抽象层过厚，Tool-Calling 已是 LLM 原生能力，自研更可控、轻量、易调试 |
| 向量库独立 vs pgvector | **pgvector** | 一期数据量不大，减少组件数量；向量和业务数据同库便于联合查询 |
| 前端重 vs 轻 | **轻量看板** | 主交互走钉钉，Web 只做可视化和管理，不需要复杂前端 |
| EspoCRM 继续用 vs 自建数据层 | **自建** | AI-Native 架构下 Agent 直接读写 PostgreSQL，无需 EspoCRM 中间层 |
| Celery vs 纯 asyncio | **Celery 做重任务** | 批量评分、报告生成等需要独立 Worker；轻量异步用 FastAPI 原生 asyncio |

### 7.8 Agent 模型部署分类（本地 vs 云端 API）

> 不是所有 Agent 都必须跑在本地 2×4090 上。
> 核心原则：**涉及内部经营数据的 Agent 必须本地推理；不涉及机密或可完全脱敏的 Agent 可调用云端大模型 API**。
> 使用云端 API 的好处：减轻本地 GPU 负载、获得更强模型能力（云端可用 72B/236B 级别模型）、降低并发瓶颈。

#### 第一类：必须本地部署（涉及内部机密数据）

这些 Agent 的 Prompt 输入中包含客户名称、交易金额、合同条款、经营指标等内部数据，**绝对不能发送到外部服务器**：

| Agent | 涉及的敏感数据 | 不能上云的原因 |
|-------|--------------|--------------|
| **Supervisor Agent** | 所有用户消息、全部业务数据 | 作为总调度，它看到一切，是数据汇聚点 |
| **销售助手 Agent** | 客户信息、联系方式、商机金额、报价 | 直接操作全部 CRM 数据，Prompt 中包含具体客户和交易细节 |
| **数据问答 Agent** | 收入、目标达成率、团队业绩、经营指标 | 查询和分析的都是核心经营数据 |
| **合同审查 Agent** | 合同条款、金额、付款条件、违约责任 | 合同是高度机密的商业文件 |
| **巡检 Agent** | 全量客户/商机/合同/回款数据 | 每日全量扫描，看到所有业务数据 |
| **预警 Agent** | 回款逾期金额、客户流失风险、目标缺口 | 涉及财务敏感阈值和客户风险评估 |
| **催收 Agent** | 应收款项、回款历史、客户信用状况 | 财务核心数据，泄露后果严重 |
| **晨报 Agent** | 每日全量经营汇总、团队销售排名 | 经营日报等于公司经营体检报告 |
| **客户洞察 Agent** | 客户画像、交易历史、决策链、竞对分析 | 包含竞争情报和客户深度分析 |
| **商机预测 Agent** | 商机金额、赢率、竞争对手、输赢原因 | 销售管线数据是核心商业机密 |
| **漏斗分析 Agent** | 管线金额分布、转化率、收入预测 | 这就是公司的营收预测，绝对机密 |
| **规划 Agent** | 销售工作量、客户优先级排序、分配策略 | 包含客户战略优先级和资源分配决策 |

> **12 个 Agent 必须本地**，全部运行在 AI 算力节点 (172.16.11.194) 的 vLLM 上。

#### 第二类：可使用云端大模型 API（不涉及机密 / 可脱敏设计）

这些 Agent 的输入**可以设计为完全脱敏**，只传递特征类别而不传递具体客户/金额信息：

| Agent | 脱敏设计 | 传给云端的数据示例 | 绝不传的数据 |
|-------|---------|------------------|------------|
| **线索评分 Agent** | 只传行业、规模区间、来源渠道、地区等类别特征 | `{行业: "教育", 规模: "100-500人", 来源: "展会", 地区: "华东", 有明确需求: true}` | 公司名称、联系人、电话、邮箱 |
| **行为分析 Agent** | 只传统计特征（匿名化的行为数据） | `{平均跟进周期: 3.2天, 周活动数: 12, 邮件响应率: 68%, 成交率: 23%}` | 销售姓名、客户名称、具体金额 |

> **2 个 Agent 可用云端 API**。推荐使用阿里云 Qwen-Plus API 或 DeepSeek API，享受更强模型能力。

#### 第三类：混合模式（主逻辑本地，子任务可调云端）

任何 Agent 在执行过程中，以下**子任务**不涉及内部数据，可以调用云端 API 获得更好的效果：

| 可云端执行的子任务 | 说明 | 调用时机 |
|------------------|------|---------|
| **行业知识查询** | "教育行业机械臂市场规模和趋势？" | 客户洞察 Agent 补充行业背景 |
| **通用话术生成** | "写一段给教育行业客户的开场白" | 销售助手 Agent 推荐话术（只传行业，不传客户名） |
| **公开数据解读** | 解读天眼查返回的工商信息 | 线索进入时自动分析（天眼查数据本身已公开） |
| **文案润色** | 把 Agent 生成的日报文字润色得更通顺 | 晨报 Agent 后处理（内容已完成，只做语言优化） |
| **合同模板起草** | "生成一份机械臂采购合同通用模板" | 合同审查 Agent 需要风格参考（只传品类，不传具体条款和金额） |
| **竞品公开分析** | "XX公司的公开产品线和市场定位" | 竞品分析子任务（分析的是公开信息） |

#### 架构实现：统一调度 + 双通道推理

```
Agent 推理请求
      │
      ▼
┌──────────────┐
│  模型路由器   │  ← 根据 Agent 类型 + 任务类型 自动选择通道
│  (Model      │
│   Router)    │
└──┬───────┬───┘
   │       │
   ▼       ▼
┌──────┐ ┌──────────────┐
│ 本地  │ │ 云端 API      │
│ vLLM │ │ (Qwen-Plus/  │
│:8000 │ │  DeepSeek)   │
└──────┘ └──────────────┘

# 路由规则 (伪代码)
def route_model(agent_name, task_type):
    # 第一类: 必须本地
    if agent_name in LOCAL_ONLY_AGENTS:
        return "local_vllm"

    # 第二类: 可用云端
    if agent_name in CLOUD_OK_AGENTS:
        return "cloud_api"

    # 第三类: 按子任务判断
    if task_type in ["industry_knowledge", "template_generation",
                     "public_data_analysis", "text_polishing"]:
        return "cloud_api"

    # 默认: 本地
    return "local_vllm"
```

#### 成本与性能收益

| 维度 | 全部本地 | 本地 + 云端混合 |
|------|---------|---------------|
| 本地 GPU 负载 | 100% Agent 请求 | ~80% Agent 请求（核心机密） |
| 云端 API 成本 | ¥0 | 约 ¥200-500/月（评分+分析+子任务） |
| 并发能力 | 受限于 2×4090 | 核心 Agent 本地 + 辅助任务云端分流 |
| 模型能力 | 30B-A3B（激活 3B） | 核心 30B-A3B + 云端可用 72B/236B 级别 |
| 延迟 | 全部低延迟 | 核心低延迟 + 云端子任务 200-500ms |

> **推荐采用混合模式**：12 个核心 Agent 本地推理保证数据安全，2 个可脱敏 Agent + 子任务调用云端 API 缓解 GPU 压力。

### 8.3 Agent 专属表设计

#### AuditEvent (审计事实流)

```
不可变的事件溯源表，记录所有业务操作的事实。
Agent 的「长期陈述性记忆」——发生过什么。

字段:
  id             BIGSERIAL PRIMARY KEY
  event_id       UUID UNIQUE           -- 事件唯一ID
  timestamp      TIMESTAMPTZ NOT NULL  -- 精确时间
  actor_type     VARCHAR(20)           -- 'user' | 'agent' | 'system'
  actor_id       VARCHAR(100)          -- 操作者ID
  agent_id       VARCHAR(50)           -- 如果是Agent操作，哪个Agent
  entity_type    VARCHAR(50)           -- 'lead' | 'customer' | 'opportunity' ...
  entity_id      UUID                  -- 实体ID
  action         VARCHAR(50)           -- 'create' | 'update' | 'delete' | 'stage_change' ...
  changes        JSONB                 -- 变更前后对比 {"field": {"old": x, "new": y}}
  context        JSONB                 -- 操作上下文 (来源、推理链路等)
  
索引: timestamp, entity_type+entity_id, actor_id
分区: 按月分区
策略: INSERT ONLY，不允许 UPDATE/DELETE
```

#### AgentLog (Agent 操作日志)

```
记录 Agent 每次推理的完整链路。
用于调试、优化和审计 Agent 行为。

字段:
  id             BIGSERIAL PRIMARY KEY
  trace_id       UUID UNIQUE           -- 追踪ID（一次完整交互）
  agent_type     VARCHAR(50)           -- 'supervisor' | 'lead_score' | 'insight' ...
  trigger_type   VARCHAR(30)           -- 'chat' | 'event' | 'schedule' | 'agent_delegate'
  trigger_detail JSONB                 -- 触发详情
  user_id        VARCHAR(100)          -- 关联用户 (如有)
  reasoning      JSONB                 -- 推理链路 [{step, thought, action, result}]
  tools_called   JSONB                 -- 工具调用记录 [{tool, params, result, duration_ms}]
  llm_calls      JSONB                 -- LLM调用 [{prompt_tokens, completion_tokens, duration_ms}]
  final_output   TEXT                  -- 最终输出
  total_duration INTEGER               -- 总耗时(ms)
  status         VARCHAR(20)           -- 'success' | 'partial' | 'error' | 'timeout'
  created_at     TIMESTAMPTZ
```

#### Knowledge (知识向量表)

```
Agent 的语义记忆，使用 pgvector 存储向量。

字段:
  id             BIGSERIAL PRIMARY KEY
  category       VARCHAR(50)           -- 'win_case' | 'lose_case' | 'product' | 'industry' | 'faq'
  title          VARCHAR(500)          -- 知识标题
  content        TEXT                  -- 原始文本
  embedding      vector(1024)          -- 向量表示 (一期: 阿里云百炼 text-embedding-v3 / 二期: 本地 BGE-M3，均为 1024 维)
  metadata       JSONB                 -- 元数据 (来源、时间、关联实体等)
  created_at     TIMESTAMPTZ
  updated_at     TIMESTAMPTZ
  
索引: embedding 使用 HNSW 索引
```

## 9. 接口设计概要

> AI-Native 架构下，API 分两类：  
> **1) Agent 对外接口** — 人/钉钉与 Agent 的交互入口  
> **2) Agent 工具接口** — Agent 内部调用的工具函数（不直接暴露给用户）

### 9.1 对外 API：Agent 交互入口

#### 核心理念：用户不需要知道有哪些 API，只需要和 Agent 对话

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/v1/agent/chat` | POST (SSE) | **主入口**：自然语言对话，流式返回。Agent 理解意图后自动执行所有操作 |
| `/api/v1/agent/chat/{session_id}` | GET | 获取对话历史 |
| `/api/v1/agent/confirm/{action_id}` | POST | 确认/拒绝 Agent 提出的高风险操作 |
| `/api/v1/dingtalk/callback` | POST | 钉钉机器人回调入口 |
| `/api/v1/dingtalk/event` | POST | 钉钉事件订阅回调 |
| `/api/v1/dingtalk/card/action` | POST | 钉钉互动卡片操作回调 |
| `/api/v1/auth/dingtalk` | POST | 钉钉 SSO 登录，返回 JWT |
| `/api/v1/auth/refresh` | POST | Token 刷新 |

#### 示例：一个 API 搞定一切

```
传统 CRM API (用户需要了解几十个接口):
  POST /api/v1/leads/           -- 创建线索
  GET  /api/v1/leads/           -- 查询线索
  PUT  /api/v1/leads/{id}       -- 更新线索
  GET  /api/v1/customers/       -- 查询客户
  POST /api/v1/opportunities/   -- 创建商机
  PUT  /api/v1/opportunities/{id}/stage  -- 推进阶段
  GET  /api/v1/analytics/funnel -- 漏斗报表
  ...（几十个接口）

AI-Native API (用户只需要一个入口):
  POST /api/v1/agent/chat
  {
    "message": "帮我查一下华东区这个月新增了多少线索",
    "session_id": "xxx"
  }

  Agent 内部自动: 理解意图 → 调用查询工具 → 聚合数据 → 返回自然语言答案
```

### 9.2 看板 API：Web 可视化 & 降级操作

> 轻量看板的数据 API，仅用于可视化展示和 AI 降级时的手动操作。

| 分组 | 路径前缀 | 说明 |
|------|---------|------|
| 仪表盘 | `/api/v1/dashboard/` | 首页看板数据（今日概览、漏斗、目标进度） |
| 线索 | `/api/v1/leads/` | 线索列表/详情/CRUD (降级用) |
| 客户 | `/api/v1/customers/` | 客户列表/详情/360视图 |
| 商机 | `/api/v1/opportunities/` | 商机列表/详情/漏斗视图 |
| 活动 | `/api/v1/activities/` | 活动时间线 |
| 分析 | `/api/v1/analytics/` | 报表/图表数据 |
| 管理 | `/api/v1/admin/` | Agent 配置/日志/健康状态 |

### 10.1 基础安全

| 层面 | 措施 |
|------|------|
| 认证 | 钉钉 SSO + JWT，Token 有效期 30 分钟，Refresh Token 7 天 |
| 授权 | RBAC (管理员/经理/销售)，数据权限按部门隔离 |
| 传输 | 内网通信；对外接口 HTTPS (Nginx SSL 终结) |
| 数据 | 敏感字段加密存储；SQL 参数化查询；API Key 脱敏 |
| 审计 | 所有操作记录到 AuditEvent 事实流，不可篡改 |

### 10.2 Agent 行为管控 (三级风险控制)

| 风险等级 | 操作类型 | Agent 行为 | 举例 |
|---------|---------|-----------|------|
| **低风险** | 查询/读取 | 自动执行，直接返回 | 查询线索列表、查看客户详情 |
| **中风险** | 单条数据修改 | 执行 + 事后通知 | 更新商机阶段、记录活动 |
| **高风险** | 批量/财务/删除 | 暂停 + 推送确认卡片 + 等待人工审批 | 批量分配线索、删除客户、修改合同金额 |

### 10.3 Agent 安全规则

```
1. 所有 Agent 生成内容必须标记: agent_id + model_version + timestamp
2. Agent 不能绕过 RBAC 权限 —— 工具调用前检查当前用户权限
3. 写操作必须幂等 —— 相同操作重复执行不产生副作用
4. 所有写操作可回滚 —— AuditEvent 记录变更前后快照
5. Agent 推理链路完整记录 —— 可审计、可回放、可追责
6. LLM Prompt 注入防护 —— 输入清洗 + Prompt 边界隔离
7. 敏感数据不进 Prompt —— 客户密码/银行卡号等不传入 LLM
```

### 10.4 AI 降级保障

| 场景 | 降级行为 |
|------|---------|
| LLM 服务不可用 | AI 字段显示"AI 暂不可用"，Web 看板 CRUD 正常使用 |
| LLM 响应超时 (>30s) | 返回缓存结果 (如有) 或提示稍后重试 |
| Agent 引擎崩溃 | 自动重启 + 钉钉通知管理员 + 纯 CRUD 模式运行 |
| AI 结果置信度过低 | 标记"低置信度"，不执行仅供参考 |
| AI 连续异常 | 自动熔断，停止该类 Agent 调用，通知管理员 |

## 12. 开发阶段部署方案

> **现实约束：4 台硬件既是开发环境，也是最终运行环境，没有独立的"生产集群"。**
> 因此部署策略不是"开发机 → 上线到生产机"，而是**同一组机器在不同阶段切换运行模式**。

### 12.1 同一硬件的两种运行模式

内网 4 台机器始终承担 §6 所定义的角色（网关/CRM/AI/数据中心），只是运行方式随项目阶段演进；第 5 台机器 (orchestrator) 承担调度 + Review + 测试 + 预发布验收：

| 维度 | 开发模式（一期开发阶段） | 运行模式（交付上线后） |
|------|------------------------|---------------------|
| 部署方式 | 直接运行 / Poetry venv，热重载优先 | Docker Compose 编排，systemd 托管 |
| Nginx | 可选，联调时启用 | 完整反向代理 + SSL + 限流 |
| PostgreSQL | 单实例，seed 脚本灌模拟数据 | 主从流复制，定期备份 |
| Redis | 单实例，默认配置 | 持久化 + AOF，密码保护 |
| vLLM | 相同参数，常驻运行 | 相同参数，常驻运行 |
| 监控 | 可选，按需启用 | Prometheus + Grafana 常驻 |
| 前端 | Vite dev server + HMR | Nginx 托管构建后的静态文件 |
| Agent 引擎 | uvicorn --reload 热重载 | gunicorn/uvicorn workers + systemd |
| 数据 | 开发测试数据，可随时重置 | 真实业务数据，不可丢失 |

> **关键认知**：vLLM 在两种模式下配置完全一致——模型加载慢、显存宝贵，没必要区分。
> 开发阶段的 vLLM 就是上线后的 vLLM，**从第一天起就是生产级运行**。

### 12.3 启动顺序（开发模式）

分 3 层依次启动，每层服务就绪后再启动下一层：

```

[AI 节点]   vLLM        → 加载模型约 2-3 分钟，启动后常驻（tmux/screen）

  ★ 这 3 个服务无论开发还是运行，配置和启动方式基本一致。
  ★ vLLM 尤其不要停，它就是生产推理服务，从第一天起常驻。

第 2 层：业务服务 — 开发时热重载，运行时 Docker/systemd
─────────────────────────────────────────
[CRM 设备]  CRM 后端     → uvicorn --reload :8900    (开发)
                          → docker compose up crm     (运行)
[AI 节点]   Agent 引擎   → uvicorn --reload :8100    (开发)
                          → docker compose up agent    (运行)

第 3 层：辅助服务 — 按需启动
─────────────────────────────────────────
[CRM 设备]  前端 dev     → Vite HMR :3000             (开发)
                          → nginx serve dist/          (运行)
[网关]      Nginx        → 联调/上线时启动
[数据中心]  Celery       → 开发异步任务时启动
[数据中心]  PG 备库      → 上线前配置流复制
[数据中心]  监控         → 上线前配置 Prometheus + Grafana

### 13.3 全自动开发循环

人不参与编码循环。Orchestrator 读取设计文档后，驱动以下全自动循环：

```
                    全自动开发循环 (Autonomous Dev Loop)
                    ══════════════════════════════════════

  ① 解析设计文档 + Sprint 任务卡
     │  Orchestrator 的 Doc Parser 读取 docs/ 和 contracts/
     │  结合 vLLM 将文档拆解为结构化编码任务
     │  按依赖关系排序，入队到 Redis 任务队列
     ▼
    ② aider 并行编码 (最多 4 台机器同时)
      │  Orchestrator 通过 SSH 在 4 台机器上启动 aider
      │  aider 通过一步 API(OpenAI兼容) 调用 Claude Opus 4.6 生成代码
     │  每个编码任务 5-15 分钟
     │  aider 自动 commit
     ▼
  ③ 三层自动 Review
     │  Layer 1: 静态检查 (py_compile + ruff)
     │  Layer 2: 契约对齐检查 (LLM 对比 contracts/)
     │  Layer 3: 设计符合度检查 (LLM 对比 docs/)
     │  不通过 → 生成修复指令 → 回到 ② (最多 3 次)
     ▼
  ④ git push → orchestrator 本地自动测试
     │  post-receive hook 触发 orchestrator 本地执行
     │  pytest 单元测试 + 云端API多模型Judge 业务评估 (GPT-4.1/Qwen-Max/DeepSeek-V3)
     │  测试报告本地直读，零网络延迟
     ▼
  ⑤ 判定
     ├─ 🟢 通过 → 取下一个任务 → 回到 ②
     ├─ 🔴 失败 → 生成修复指令 → 回到 ② (最多 3 次重试)
     └─ ⚠️ 3 次仍失败 → 钉钉通知人工介入
```

- **单个任务周期**: 5-15 分钟
- **每天可完成**: 50-100 个独立编码任务
- **运行方式**: 7×24 持续运行（白天高优先级任务，夜间重构/补测试）

### 13.4 接口契约：5 台机器的同步枢纽

5 台机器并行编码的关键是**接口对齐**。解决方案不变：**contracts/ 目录作为单一真相源**。

```

├── contracts/                    ← 接口契约目录 (你维护)
│   ├── agent-api.yaml            ← Agent 引擎对外接口定义

│   ├── agent-tools.yaml          ← Agent Tool Calling Schema
│   ├── event-bus.yaml            ← Redis Stream 事件格式
│   └── db-schema.sql             ← 数据库表结构 (单一真相源)
├── orchestrator/                 ← Orchestrator 薄调度层 (orchestrator 专用服务器)
├── agent/                        ← 4090 aider 自动编码
├── crm/                          ← mac_min_8T aider 自动编码
├── frontend/                     ← mac_min_8T aider 自动编码
├── deploy/                       ← gateway + data_center aider 自动编码
├── scripts/                      ← data_center aider 自动编码

```

**自动化流程中的契约使用**：

```
1. 你在 contracts/ 中定义或更新接口契约
   → git push

2. Orchestrator 检测到 contracts/ 变更
   → 自动为相关机器生成对齐编码任务
   → 例如: contracts/crm-api.yaml 新增接口 → W2 自动生成对应 API

3. aider 编码时自动加载 contracts/ 文件作为上下文
   → 保证生成的代码与契约签名一致

4. AutoReviewer Layer 2 自动对比代码与契约
   → 不一致自动修正
```

### 13.7 自动 Review 机制（三层检查）

人工 Review 被 LLM 驱动的自动三层 Review 完全替代：

```
  Layer 1: 静态检查 (0 成本, 秒级)
  ├── python -m py_compile  ← 语法正确性
  ├── ruff check            ← 代码规范
  └── 结果: 有错 → 直接生成修复指令

  Layer 2: 契约对齐检查 (LLM 驱动, ~30s)
  ├── 对比代码与 contracts/*.yaml 的接口签名
  └── 结果: 不一致 → 生成对齐修复指令

  Layer 3: 设计符合度检查 (LLM 驱动, ~60s)
  ├── 对比代码与 docs/04-系统概要设计.md 的设计意图
  ├── 评分: 功能完整性 / 接口正确性 / 错误处理 / 代码质量 / 可运行性
  └── 结果: < 4.0/5.0 分 → 生成具体修改建议 → 重试

  三层全通过 → git push → orchestrator 本地自动测试
```

| **联调 + 调试** | 40h | 12h | 5h (自动Review+自动测试大幅减少联调) |
| **Review** | — | 15h (人工巡回) | 0h (三层自动Review) |
| **测试 + 评估** | 20h | 3h | 2h (全自动) |
| **总计** | **178h** | **~61.5h** | **~24h (约 3 个工作日)** |

> **全自动模式的核心优势**：
> 1. **消除人工瓶颈** — Review 不再是串行瓶颈，三层自动检查比人工更严格
> 2. **7×24 运行** — 不受人类工作时间限制，夜间也在编码
> 3. **自动重试** — 编码失败自动修复，不需等人排查
> 4. **一致性** — 所有代码都经过统一的静态+契约+设计三层检查
> 一期 8 周时间非常充裕，关键路径仍是 **Agent Prompt 效果调优**。

### B. 术语表

#### 基础概念

| 术语 | 说明 |
|------|------|
| **CRM** | Customer Relationship Management，客户关系管理系统。帮助企业管理客户信息、销售流程、售后服务的软件 |
| **CRUD** | Create/Read/Update/Delete 的缩写，即数据的「增删改查」四种基本操作 |
| **API** | Application Programming Interface，应用编程接口。软件之间相互通信的标准化方式，类似「插座接口」 |
| **RESTful API** | 一种 API 设计风格，用 HTTP 请求（GET/POST/PUT/DELETE）对应数据的查/增/改/删 |
| **SSO** | Single Sign-On，单点登录。用一个账号（如钉钉）登录所有系统，不用重复输密码 |
| **JWT** | JSON Web Token，一种登录凭证格式。用户登录后获得一个加密令牌，后续请求携带它证明身份 |
| **RBAC** | Role-Based Access Control，基于角色的权限控制。不同角色（管理员/经理/销售）看到不同数据、执行不同操作 |
| **SDK** | Software Development Kit，软件开发工具包。别人提供的现成代码库，拿来直接用 |

#### AI / 大模型相关

| 术语 | 说明 |
|------|------|
| **LLM** | Large Language Model，大语言模型。能理解和生成自然语言的 AI 模型，如 ChatGPT、Qwen 等 |
| **Agent** | AI 智能体。不只是回答问题，还能自主规划、调用工具、执行操作的 AI 程序 |
| **AI-Native** | 以 AI Agent 为系统核心（而非附加层）的架构模式。Agent 就是系统本身，而不是系统上装了一个 AI 插件 |
| **Supervisor Agent** | 总调度 Agent，接收用户消息后理解意图、分解任务、编排执行，相当于 Agent 团队的「项目经理」 |
| **Tool Calling** | 工具调用。LLM 在对话中识别出需要执行的操作，自动调用对应的函数（如查数据库、发消息），而不只是生成文字 |
| **Prompt** | 提示词。给 LLM 的指令和上下文，决定了 LLM 如何思考和回答。设计好的 Prompt 是 Agent 能力的关键 |
| **Token** | LLM 处理文本的最小单位，约等于 1-2 个中文字或 3-4 个英文字母。LLM 的输入输出都按 Token 计量 |
| **MoE** | Mixture of Experts，混合专家架构。模型虽然总参数很大（30B），但每次只激活一小部分（3B），所以推理速度快、显存占用低 |
| **vLLM** | 一个高性能 LLM 推理引擎，能高效利用 GPU 同时处理多个请求，是我们用来运行 Qwen 模型的软件 |
| **Continuous Batching** | 连续批处理。vLLM 的核心技术，请求随到随处理，不用等凑满一批，大幅降低延迟 |
| **PagedAttention** | vLLM 的显存管理技术，像操作系统管理内存一样管理 GPU 显存，让多个请求共享显存，提高并发能力 |
| **Prefix Caching** | 前缀缓存。多个请求如果开头相同（如同样的系统提示词），GPU 里只计算一次，后续复用，节省时间 |
| **Tensor Parallel** | 张量并行。把一个模型拆到多张 GPU 卡上同时计算，加快推理速度。我们用 2 张 4090 并行 |
| **Embedding** | 向量嵌入。把文本转换成一串数字（向量），数字相近的文本含义也相近。用于知识库的语义搜索 |
| **向量数据库** | 专门存储和检索向量的数据库。输入一段文字，能找到含义最相近的其他内容 |
| **pgvector** | PostgreSQL 的向量搜索扩展插件，让普通数据库也能做向量检索，不用额外装一个独立的向量数据库 |
| **RAG** | Retrieval-Augmented Generation，检索增强生成。Agent 先从知识库检索相关内容，再结合检索结果生成回答，提高准确性 |
| **NLU** | Natural Language Understanding，自然语言理解。让 AI 理解人说的话是什么意思（识别意图和关键信息） |
| **SSE** | Server-Sent Events，服务器推送事件。一种流式传输技术，让 Agent 的回答像打字一样逐字显示，而不是等全部生成完才一次返回 |

#### 架构相关

| 术语 | 说明 |
|------|------|
| **Supervisor** | 见 Supervisor Agent |
| **结构化记忆** | Agent 的长期记忆，存在 PostgreSQL 里。就是 CRM 的业务数据：客户、商机、合同等 |
| **语义记忆** | Agent 的知识库，通过向量存储。存放：赢单经验、产品知识、行业知识等，支持按含义检索 |
| **工作记忆** | Agent 的短期记忆，存在 Redis 里。包括：当前对话上下文、待确认的操作、正在执行的任务状态 |
| **事件总线** | Agent 的神经网络。所有业务操作产生事件 → 事件总线广播 → 关心这个事件的 Agent 自动触发。用 Redis Stream 实现 |
| **审计事实流** | 不可变的操作日志。记录「谁在什么时间对什么数据做了什么操作」，只增不改不删，用于审计和回溯 |
| **事件驱动** | 一种架构模式：系统中发生的事情（如创建线索）产生「事件」，其他模块订阅感兴趣的事件自动响应，而不是互相直接调用 |
| **降级** | 当 AI 服务出故障时，系统自动切换到无 AI 的基本模式继续运行（如用 Web 表单做 CRUD），保证业务不中断 |
| **幂等** | 同一个操作执行一次和执行多次效果相同。比如「把商机阶段改为方案评审」执行两次，结果应该和执行一次一样 |

#### 业务术语

| 术语 | 说明 |
|------|------|
| **线索 (Lead)** | 潜在的销售机会信息，尚未确认是否有真实需求。比如展会扫码留下联系方式的人 |
| **客户 (Customer/Account)** | 已确认有合作意向或已成交的企业/个人 |
| **联系人 (Contact)** | 客户企业中的具体对接人，一个客户可能有多个联系人 |

| **漏斗 (Funnel)** | 销售漏斗，把销售流程可视化为漏斗形状：大量线索 → 较少客户 → 少量商机 → 成交。每层的转化率是核心指标 |
| **赢率** | 商机最终成交的概率预测，由 Agent 基于多维度数据计算 |
| **客户画像** | 对一个客户的全方位描述：基本信息、交易历史、互动记录、偏好特征、健康度等 |

| **决策链** | 客户内部的采购决策关系：谁是最终决策者、谁是影响者、谁是使用者。搞清决策链是赢单关键 |
| **线索池** | 未分配给具体销售人员的线索集合，销售可以从池中领取或由系统自动分配 |
