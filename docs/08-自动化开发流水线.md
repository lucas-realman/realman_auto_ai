## 1. 流水线总体架构与理念

### 1.1 范式转换：从「人指挥 Copilot」到「机器自治」

旧模式（v1.0）中，你打开 5 个 VS Code 窗口，逐窗口给 Copilot 下达指令、巡回 Review、手动 push/pull。
你是整个流水线的 **瓶颈**——4 个窗口的产出速度受限于你切换 Review 的速度。

新模式（v2.0）中，你的角色变为 **系统方案设计者**：

```
┌──────────────────────────────────────────────────────────────────────────┐
│                                                                          │
│  旧模式 (v1.0):  你 ──指令──▶ Copilot ──代码──▶ 你 Review ──▶ git push │
│                  ↑                                    │                  │
│                  └────────── 人工循环 ────────────────┘                  │
│                                                                          │
│  新模式 (v2.0):  你 ──设计文档──▶ Orchestrator ──自动──▶ 5台机器协作    │
│                                     ↑        自动编码 → 自动Review       │
│                                     │        → 自动测试 → 自动迭代       │
│                                     │        → 自动部署                  │
│                                     └── 只在异常时通知你 ──              │
│                                                                          │
└──────────────────────────────────────────────────────────────────────────┘
```

**你的唯一职责**：编写/更新 `docs/` 和 `contracts/` 中的设计文档与接口契约。
**机器的全部职责**：读取文档 → 拆解任务 → 编码 → Review → 测试 → 迭代修复 → 部署 → 报告。

### 1.6 人工介入的触发条件

流水线设计为**全自动**，但以下情况会通过钉钉通知人工介入：

| # | 触发条件 | 通知内容 | 人工操作 |
|---|---------|---------|---------|
| 1 | 编码任务连续 3 次 Review 失败 | 失败原因 + 相关代码片段 | 修改设计文档/契约，或手动修复代码 |
| 2 | 测试评分低于基线（多模型加权评分 < 3.0） | 失败用例 + Agent 回答对比 + 各Judge评分明细 | 调整 Prompt 设计或业务逻辑 |
| 3 | 硬件故障（SSH 不通、GPU 异常） | 故障机器 + 错误日志 | 物理排查/重启 |
| 4 | Git 冲突（串行提交下极低概率） | 冲突文件列表 | 手动解决冲突 |
| 5 | Sprint 里程碑检查点 | Sprint 完成度报告 | 确认是否进入下一 Sprint |
| 6 | 设计文档与代码实现偏差 > 30% | 偏差分析报告 | 更新设计文档或回退代码 |

> **原则**：通知是为了让你**知道**并**决策**，不是让你**操作**。
> 大部分情况下你只需在钉钉回复「继续」或「暂停」。

## 2. Orchestrator 核心设计

> **关键认知：aider 已有能力 vs Orchestrator 职责边界**
>
> aider 本身已是一个完整的自动编码循环：读取上下文文件 → LLM 生成代码 → 自动写入文件 → git commit → 运行 lint/test → 测试失败自动生成修复指令 → 重试直到通过或达到上限。因此 **Orchestrator 不需要从零实现编码、Review、测试、修复等能力**，只需要做 aider 做不到的事：**跨机器调度 + 任务编排 + 结果汇总 + 通知**。
>
> 实际实现路径：
> - **S1 Day 1-2**：编写 `orchestrator.sh`（~200 行），能 SSH 到 5 台机器运行 aider → 自动化立即可用
> - **S3-4 渐进增强**：加入 DocParser（云端 API 一次调用）、钉钉通知、多模型 Judge
> - **S5+**：按需加入 Redis 任务队列、并发调度等高级功能
>
> 以下伪代码展示**逻辑架构**（各模块的职责和数据流），实际实现会比伪代码简单得多，因为 aider 已覆盖编码→测试→修复的核心循环。

Orchestrator 是整条流水线的调度层，运行在专用服务器 orchestrator (172.16.14.201, Xeon 64C + 128G + 2×4090)，以 Python/Shell 脚本方式运行。AutoReview 和 DocParser 走云端 API（强模型更适合代码审查和文档解析），Agent 集成测试推理统一调用 4090 上的 vLLM A（内网延迟 <1ms，测试环境与生产推理配置完全一致）。orchestrator 的 2×4090 GPU 释放给其他计算任务。pytest 在本机执行，编码→测试反馈全内网闭环。

### 2.2 核心模块说明

#### 2.2.1 Doc Parser（文档解析器）

从 Git 仓库读取设计文档和 Sprint 任务卡，解析为结构化任务列表：

```python
# orchestrator/doc_parser.py — 伪代码

class DocParser:
    """解析设计文档 + Sprint 任务卡 → 生成编码任务"""

    def __init__(self, repo_path: str, llm_client):
        self.repo_path = repo_path
        self.llm = llm_client  # 连接 4090 上的 vLLM

    def parse_sprint_tasks(self, sprint_id: int) -> list[CodingTask]:
        """
        读取 07-Sprint任务卡.md 中指定 Sprint 的内容，
        结合 04-系统概要设计.md 中的架构描述，
        生成可执行的编码任务列表。
        """
        sprint_card = self._read_doc(f"docs/07-Sprint任务卡.md")
        design_doc = self._read_doc(f"docs/04-系统概要设计.md")
        contracts = self._read_contracts()

        prompt = f"""
        你是一个任务拆解引擎。请根据以下文档，将 Sprint {sprint_id} 的任务
        拆解为独立的编码任务。

        每个任务需包含:
        - task_id: 唯一标识
        - target_machine: 目标机器 (4090/mac_min_8T/gateway/data_center)
        - target_dir: 目标目录
        - description: 任务描述 (给 aider 的详细指令)
        - context_files: 需要 aider 读取的上下文文件
        - depends_on: 依赖的前置任务 ID 列表
        - acceptance: 自动验收标准 (用于 Review 判定)

        === Sprint 任务卡 ===
        {sprint_card}

        === 系统设计 (相关章节) ===
        {design_doc}

        === 接口契约 ===
        {contracts}
        """
        tasks = self.llm.generate(prompt, response_format="json")
        return [CodingTask(**t) for t in tasks]
```

#### 2.2.2 Task Engine（任务引擎）

管理任务队列、依赖关系和执行顺序：

```python
# orchestrator/task_engine.py — 伪代码

class TaskEngine:
    """任务调度引擎：维护队列、处理依赖、管理重试"""

    def __init__(self, redis_client):
        self.queue = redis_client       # Redis 作为任务队列
        self.max_retries = 3            # 每个任务最多重试 3 次
        self.concurrent_limit = 4       # 最多 4 台机器同时编码

    def enqueue(self, tasks: list[CodingTask]):
        """将任务入队，按依赖关系排序"""
        ordered = self._topological_sort(tasks)
        for task in ordered:
            self.queue.rpush("task:pending", task.to_json())

    def next_batch(self) -> list[CodingTask]:
        """
        取出下一批可并行执行的任务：
        - 无未完成的前置依赖
        - 目标机器不冲突（每台机器同时只执行 1 个任务）
        - 最多 4 个任务并行
        """
        pending = self._get_pending()
        completed = self._get_completed_ids()
        batch = []
        used_machines = set()

        for task in pending:
            if all(dep in completed for dep in task.depends_on):
                if task.target_machine not in used_machines:
                    batch.append(task)
                    used_machines.add(task.target_machine)
            if len(batch) >= self.concurrent_limit:
                break
        return batch

    def handle_result(self, task_id: str, result: TaskResult):
        """处理任务结果：通过/重试/升级"""
        if result.passed:
            self._mark_completed(task_id)
        elif result.retry_count < self.max_retries:
            self._requeue_with_fix(task_id, result.fix_instruction)
        else:
            self._escalate(task_id, result)  # 钉钉通知人工
```

#### 2.2.3 Dispatcher（SSH 分发器）

通过 SSH 在远程机器上执行 aider 编码任务：

```python
# orchestrator/dispatcher.py — 伪代码

import asyncio
import asyncssh

class Dispatcher:
    """SSH 分发器：将编码任务推送到目标机器执行"""

    SSH_CONFIG = {
        "4090":         {"host": "172.16.11.194", "user": "user"},
        "mac_min_8T":   {"host": "172.16.12.50",  "user": "edge_sale"},
        "gateway":      {"host": "172.16.14.215", "user": "realman"},
        "data_center":  {"host": "172.16.14.90",  "user": "realman"},
    }

    async def dispatch_batch(self, tasks: list[CodingTask]) -> list[TaskResult]:
        """并行分发一批任务到不同机器"""
        coros = [self._execute_on_remote(task) for task in tasks]
        return await asyncio.gather(*coros)

    async def _execute_on_remote(self, task: CodingTask) -> TaskResult:
        """SSH 到目标机器，启动 aider 执行编码任务"""
        cfg = self.SSH_CONFIG[task.target_machine]

        async with asyncssh.connect(cfg["host"], username=cfg["user"]) as conn:
            # 1. 确保代码最新
            await conn.run("cd ~/ai-crm && git pull --rebase")

            # 2. 写入任务指令文件
            instruction = self._build_aider_instruction(task)
            await conn.run(f"cat > /tmp/task_{task.task_id}.md << 'TASK_EOF'\n"
                          f"{instruction}\n"
                          f"TASK_EOF")

            # 3. 执行 aider
            result = await conn.run(
                f"cd ~/ai-crm && "
                f"aider --model openai/claude-opus-4-6 "
                f"--openai-api-base {os.environ['OPENAI_API_BASE']} "
                f"--openai-api-key {os.environ['OPENAI_API_KEY']} "
                f"--message-file /tmp/task_{task.task_id}.md "
                f"--yes-always "        # 自动确认所有操作
                f"--auto-commits "      # 自动 commit
                f"--no-suggest-shell-commands "
                f"{' '.join(task.context_files)}",
                timeout=600             # 单任务最长 10 分钟
            )

            # 4. 收集结果
            return TaskResult(
                task_id=task.task_id,
                stdout=result.stdout,
                exit_code=result.exit_code,
                files_changed=self._parse_changed_files(result.stdout),
            )
```

### 2.3 任务状态机

每个编码任务经历以下状态流转：

```
                    任务状态机 (Task State Machine)
                    ══════════════════════════════

  ┌─────────┐    解析完成    ┌──────────┐    依赖满足    ┌─────────────┐
  │ CREATED │──────────────▶│ QUEUED   │──────────────▶│ DISPATCHED  │
  └─────────┘               └──────────┘            ┌──▶└──────┬──────┘
                                                    │         │
                                                    │    aider 执行中
                                                    │         │
                                                    │         ▼
  ┌─────────┐    Review通过  ┌──────────┐    编码完成 │   ┌─────────────┐
  │ TESTING │◀──────────────│ REVIEW   │◀───────────┼───│ CODING_DONE │
  └────┬────┘               └─────┬────┘            │   └─────────────┘
       │                          │                 │
       │ 测试结果                  │ Review 失败      │ 重新编码
       ▼                          │ (review_retry<3) │
  ┌─────────┐                     ▼                 │
  │ JUDGING │              ┌──────────────┐         │
  └────┬────┘              │ RETRY        │─────────┘
       │                   │ (带修复指令) │
  ┌────┴─────┐             └──────────────┘
  │          │                    ▲
  │    ┌─────────┐                │ 测试失败 (test_retry < 3)
  │    │ PASSED  │                │ 附带: 失败日志 + 修复指令
  │    └─────────┘                │
  │                               │
  │    ┌─────────┐                │
  └───▶│ FAILED  │────────────────┘
       └────┬────┘
            │ 累计重试 >= 3
            ▼
       ESCALATED (通知人工)
```

**关键变更**：测试失败（`FAILED`）不再直接升级为人工处理，而是将失败日志 + 错误堆
栈 + 自动生成的修复指令一起注入 `RETRY` 状态，由 aider 重新编码修复后再走
Review → 测试 完整流程，直至累计重试次数（review\_retry + test\_retry）≥ 3 才升级。

### 2.4 Orchestrator 主循环

```python
# orchestrator/main.py — 主循环伪代码

class Orchestrator:
    """主调度循环：持续运行，自动执行开发流水线"""

    def __init__(self):
        self.parser = DocParser(repo_path="~/ai-crm", llm_client=cloud_api_client)  # 云端API
        self.engine = TaskEngine(redis_client=redis)
        self.dispatcher = Dispatcher()
        self.reviewer = AutoReviewer(llm_client=cloud_api_client)   # §4 详述，云端API
        self.reporter = Reporter(dingtalk_webhook=WEBHOOK_URL)  # §6 详述

    async def run_sprint(self, sprint_id: int):
        """执行一个完整 Sprint 的自动化开发"""
        # 1. 解析 Sprint 任务
        tasks = self.parser.parse_sprint_tasks(sprint_id)
        self.engine.enqueue(tasks)
        self.reporter.notify(f"Sprint {sprint_id} 启动，共 {len(tasks)} 个任务")

        # 2. 持续执行直到所有任务完成
        while not self.engine.all_done():
            # 取下一批可并行的任务
            batch = self.engine.next_batch()
            if not batch:
                await asyncio.sleep(30)  # 等待依赖完成
                continue

            # 3. 分发执行
            results = await self.dispatcher.dispatch_batch(batch)

            # 4. 自动 Review
            for result in results:
                review = await self.reviewer.review(result)
                if review.passed:
                    # 5. git push → 触发 orchestrator 本地自动测试
                    await self._push_and_test(result)
                else:
                    self.engine.handle_result(result.task_id, review)

            # 6. 每轮报告
            self.reporter.round_summary(batch, results)

        # 7. Sprint 完成
        self.reporter.sprint_summary(sprint_id)

    async def run_continuous(self):
        """持续运行模式：监听文档变更，自动触发任务"""
        while True:
            if self._docs_changed():
                # 文档更新 → 重新解析 → 增量生成新任务
                new_tasks = self.parser.parse_incremental()
                self.engine.enqueue(new_tasks)

            # 执行待处理的任务
            batch = self.engine.next_batch()
            if batch:
                results = await self.dispatcher.dispatch_batch(batch)
                for result in results:
                    review = await self.reviewer.review(result)
                    self.engine.handle_result(result.task_id, review)

            await asyncio.sleep(10)  # 10 秒轮询
```

## 3. 自动化编码引擎

每台开发机器上运行 **aider**（AI 编码 CLI），通过 OpenAI 兼容 API 连接 Claude Opus 4.6，实现自动代码生成与提交。AutoReview 和 DocParser 走云端 API（强模型更适合代码审查和文档解析），Agent 集成测试推理统一调用 4090 上的 **vLLM 实例 A**（内网 <1ms）。

### 3.1 aider 简介与选型理由

|------|-------|-----------|-------------|

| 上下文管理 | ✅ 自动分析 repo map | 需手动指定 | 有限 |
| 自动 Git commit | ✅ 内置 | 需自行实现 | ❌ |

| 无交互模式 | ✅ `--yes-always` | 天然 | ❌ 需交互 |
| 社区活跃度 | ✅ 70k+ stars | - | - |

> aider 是目前最成熟的 AI 编码 CLI，原生支持 OpenAI 兼容 API，可直接连接 vLLM。

```python
def _build_aider_instruction(self, task: CodingTask) -> str:
    """将结构化任务转换为 aider 可理解的自然语言指令"""

    instruction = f"""
# 编码任务 {task.task_id}

## 目标
{task.description}

## 关键约束
- 严格遵循 contracts/ 中的接口定义，函数签名、参数名、返回值类型必须一致

- import 语句完整，确保代码可独立运行
- 同时生成对应的 pytest 测试文件

## 验收标准
{chr(10).join(f'- {a}' for a in task.acceptance)}

## 代码风格
- Python: PEP 8, type hints, async/await
- 缩进: 4 空格
- 字符串: 优先 f-string
- 日志: 使用 loguru
"""
    return instruction

def _build_aider_command(self, task: CodingTask) -> str:
    """构建完整的 aider CLI 命令"""
    context_args = " ".join(task.context_files)
    return (
        f"cd ~/ai-crm && "
        f"aider "
        f"--message-file /tmp/task_{task.task_id}.md "
        f"--yes-always "
        f"--auto-commits "
        f"--no-suggest-shell-commands "
```

## 4. 自动 Review 与迭代循环

编码完成后，代码不直接推送到主仓库，而是先经过 **LLM 自动 Review**。Review 对照设计文档和接口契约检查代码质量，不通过则自动生成修复指令并重试。

### 4.1 自动 Review 的三层检查

```
                    自动 Review 三层检查
                    ══════════════════

  Layer 1: 静态检查 (0 成本, 秒级)
  ├── python -m py_compile *.py        ← 语法正确性
  ├── ruff check --select E,W,F        ← 代码规范
  ├── mypy --ignore-missing-imports    ← 类型检查 (宽松模式)
  └── 结果: 有错 → 直接生成修复指令回到编码阶段

  Layer 2: 契约对齐检查 (LLM 驱动)
  ├── 对比代码与 contracts/*.yaml 的接口签名
  ├── 检查函数名、参数名、返回类型是否一致
  ├── 检查缺失的接口方法
  └── 结果: 不一致 → 生成具体的对齐修复指令

  Layer 3: 设计符合度检查 (LLM 驱动)
  ├── 对比代码与 docs/04-系统概要设计.md 的设计意图
  ├── 检查核心逻辑是否正确实现
  ├── 检查是否有硬编码、缺少异常处理、缺少日志
  ├── 检查是否有遗漏的功能点
  └── 结果: 评分 + 具体修改建议
```

### 4.2 AutoReviewer 实现

```python
# orchestrator/reviewer.py — 自动 Review 引擎

class AutoReviewer:
    """三层自动代码 Review"""

    def __init__(self, llm_client, repo_path: str):
        self.llm = llm_client
        self.repo_path = repo_path

    async def review(self, result: TaskResult, task: CodingTask) -> ReviewResult:
        """对编码结果执行三层 Review"""

        # Layer 1: 静态检查
        static_issues = await self._static_check(result.files_changed)
        if static_issues:
            return ReviewResult(
                passed=False,
                layer="static",
                issues=static_issues,
                fix_instruction=self._static_fix_instruction(static_issues)
            )

        # Layer 2: 契约对齐
        contract_issues = await self._contract_check(result, task)
        if contract_issues:
            return ReviewResult(
                passed=False,
                layer="contract",
                issues=contract_issues,
                fix_instruction=contract_issues  # LLM 已生成修复指令
            )

        # Layer 3: 设计符合度
        design_review = await self._design_check(result, task)
        if design_review.score < 4.0:
            return ReviewResult(
                passed=False,
                layer="design",
                issues=design_review.issues,
                fix_instruction=design_review.fix_instruction,
                score=design_review.score
            )

        return ReviewResult(passed=True, score=design_review.score)

    async def _static_check(self, files: list[str]) -> list[str]:
        """Layer 1: 静态检查"""
        issues = []
        for f in files:
            if f.endswith(".py"):
                # 编译检查
                result = await self._run_remote(
                    f"cd ~/ai-crm && python -m py_compile {f}"
                )
                if result.exit_code != 0:
                    issues.append(f"编译错误 {f}: {result.stderr}")

                # ruff 检查
                result = await self._run_remote(
                    f"cd ~/ai-crm && ruff check {f} --select E,W,F"
                )
                if result.exit_code != 0:
                    issues.append(f"Lint 问题 {f}: {result.stdout}")
        return issues

    async def _contract_check(self, result: TaskResult, task: CodingTask) -> str:
        """Layer 2: 对照接口契约检查"""
        code_content = self._read_files(result.files_changed)
        contracts = self._read_contracts(task)

        prompt = f"""
你是一个接口契约审查员。请对比以下代码和接口契约，检查是否一致。

## 接口契约
{contracts}

## 生成的代码
{code_content}

## 检查项
1. 函数/方法签名是否和契约定义一致（名称、参数、返回类型）？
2. 是否有契约中定义但代码中未实现的接口？
3. 参数的校验逻辑是否和契约中的约束一致（required、enum、type）？

## 输出格式
如果完全一致，返回: {{"passed": true}}
如果有不一致，返回:
{{
    "passed": false,
    "issues": ["问题描述1", "问题描述2"],
    "fix_instruction": "具体的修复指令，告诉 aider 如何修改"
}}
"""
        result = self.llm.generate(prompt, response_format="json")
        if not result["passed"]:
            return result["fix_instruction"]
        return None

    async def _design_check(self, result: TaskResult, task: CodingTask) -> DesignReview:
        """Layer 3: 对照设计文档检查"""
        code_content = self._read_files(result.files_changed)
        design_doc = self._read_relevant_design_sections(task)

        prompt = f"""
你是一个高级代码审查员。请根据设计文档评审以下代码。

## 设计文档 (相关章节)
{design_doc}

## 编码任务
{task.description}

## 验收标准
{chr(10).join(task.acceptance)}

## 生成的代码
{code_content}

## 评审维度 (每项 1-5 分)
1. **功能完整性**: 是否完整实现了设计文档要求的所有功能点？
2. **接口正确性**: 函数签名、参数、返回值是否符合契约？
3. **错误处理**: 是否有完整的异常处理？错误信息是否有上下文？
4. **代码质量**: 是否有硬编码？docstring 是否完整？日志是否充分？
5. **可运行性**: import 是否齐全？是否可以独立运行/测试？

## 输出格式 (JSON)
{{
    "scores": {{"功能完整性": 4, "接口正确性": 5, ...}},
    "average_score": 4.2,
    "issues": ["问题1", "问题2"],
    "fix_instruction": "如果平均分<4.0，给出具体修复指令"
}}
"""
        return self.llm.generate(prompt, response_format="json")
```

### 4.4 修复指令的生成策略

```python
# orchestrator/reviewer.py — 修复指令生成

def generate_fix_instruction(self, task: CodingTask,
                              review: ReviewResult,
                              retry_count: int) -> str:
    """根据 Review 结果和重试次数生成递进式修复指令"""

    if retry_count == 0:
        # 第 1 次修复：精准修复 Review 发现的问题
        return f"""

{chr(10).join(f'- {issue}' for issue in review.issues)}

只修改有问题的部分，不要重写整个文件。

"""

    elif retry_count == 1:
        # 第 2 次修复：带更多上下文和示例
        return f"""
上一次修复不完整。请重新检查以下问题：

{chr(10).join(f'- {issue}' for issue in review.issues)}

{self._read_relevant_contract(task)}

- 返回值结构必须和契约定义的 schema 匹配
- 所有 required 字段都要有校验
"""

    else:
        # 第 3 次修复：最后机会，提供完整重写指引
        return f"""
这是最后一次自动修复机会。之前的修复均未完全解决问题。

遗留问题：
{chr(10).join(f'- {issue}' for issue in review.issues)}

请考虑重新实现相关函数。以下是完整的参考信息：

设计要求：
{task.description}

接口契约：
{self._read_relevant_contract(task)}

验收标准：
{chr(10).join(f'- {a}' for a in task.acceptance)}

如果无法一次性解决所有问题，优先确保：
1. 代码可编译运行
2. 接口签名和契约一致
3. 核心逻辑正确
"""
```

### 5.2 自动 commit + push 流程

```python
# orchestrator/git_ops.py — Git 自动化操作

class GitOps:
    """Git 自动化：commit, push, pull, tag"""

    def __init__(self, dispatcher: Dispatcher):
        self.dispatcher = dispatcher

    async def push_from_machine(self, machine: str, task: CodingTask):
        """
        从指定机器 push 代码到中央仓库。
        
        策略：串行提交 + 人工确认（R-013 决议）
        - aider 编码后自动 commit，但 push 需人工确认
        - 共享文件（requirements.txt、config.py）仅在 mac_min_8T 上修改
        - 按子目录隔离（agent/ / crm/ / deploy/），冲突概率极低
        """
        cfg = Dispatcher.SSH_CONFIG[machine]
        async with asyncssh.connect(cfg["host"], username=cfg["user"]) as conn:
            # aider 已经自动 commit，push 前通知人工确认
            await self.dispatcher.notify(
                f"🔔 {machine} 编码完成，请确认后 push",
                task_summary=task.description
            )
            result = await conn.run(
                "cd ~/ai-crm && git push origin main",
                timeout=60
            )
            if result.exit_code != 0:
                # push 失败 → 通知人工处理（不自动 rebase）
                await self.dispatcher.notify(
                    f"⚠️ {machine} push 失败，请手动处理",
                    error=result.stderr
                )

    async def sync_all_machines(self):
        """所有机器 pull 最新代码"""
        coros = []
        for machine in ["4090", "mac_min_8T", "gateway", "data_center"]:
            coros.append(self._pull_on_machine(machine))
        await asyncio.gather(*coros)

    async def auto_tag_sprint(self, sprint_id: int):
        """Sprint 结束时自动打 tag"""
        version = f"v0.{sprint_id // 2 + 1}.0"
        async with asyncssh.connect(
            "172.16.12.50", username="edge_sale"
        ) as conn:
            await conn.run(
                f"cd ~/ai-crm && "
                f"git tag -a {version} -m 'Sprint {sprint_id} 完成' && "
                f"git push origin {version}"
            )

    # 注：原有 _handle_push_failure 方法（自动 rebase + GitConflictError）已移除
    # R-013 决议：1 人团队采用串行提交 + 人工确认，不需要自动冲突处理
    # 如 push 失败，通知开发者手动处理即可
```

### 5.3 post-receive Hook（测试自动触发）

Git 裸仓库的 post-receive hook 自动触发 orchestrator 本地测试：

```bash
#!/bin/bash
# /opt/git/ai-crm.git/hooks/post-receive
# 代码推送后自动触发测试

set -e

# 读取推送信息
while read oldrev newrev refname; do
    branch=$(git rev-parse --symbolic --abbrev-ref $refname)

    if [ "$branch" = "main" ]; then
        echo "=== main 分支更新，触发自动化流水线 ==="

        # 1. 获取变更文件列表
        CHANGED_FILES=$(git diff --name-only $oldrev $newrev)
        echo "变更文件: $CHANGED_FILES"

        # 2. 判断变更类型
        HAS_CODE_CHANGE=false
        HAS_DOC_CHANGE=false

        echo "$CHANGED_FILES" | grep -qE "^(agent|crm|frontend|deploy|scripts)/" && HAS_CODE_CHANGE=true
        echo "$CHANGED_FILES" | grep -qE "^(docs|contracts)/" && HAS_DOC_CHANGE=true

        # 3. 代码变更 → 触发 orchestrator 本地测试
        if [ "$HAS_CODE_CHANGE" = true ]; then
            echo "代码变更，触发 orchestrator 本地自动测试..."
            ssh orchestrator "cd ~/ai-crm && \
                git pull && \
                nohup bash scripts/run_tests.sh > logs/test_$(date +%Y%m%d_%H%M%S).log 2>&1 &"
        fi

        # 4. 文档变更 → 通知 Orchestrator 重新解析
        if [ "$HAS_DOC_CHANGE" = true ]; then
            echo "文档变更，通知 Orchestrator..."
            ssh data_center "curl -s http://localhost:9500/api/docs-changed \
                -H 'Content-Type: application/json' \
                -d '{\"files\": \"$CHANGED_FILES\"}'"
        fi

        # 5. 同步到所有开发机
        for host in 4090 mac_min_8T gateway data_center; do
            ssh $host "cd ~/ai-crm && git pull --rebase" &
        done
        wait
        echo "=== 所有机器已同步 ==="
    fi
done
```

### 6.1 三级报告体系

```
                    报告体系
                    ════════

  Level 1: 实时报告 (每批次 ~15min)
  ├── 本批 4 个任务的执行结果
```

### 6.4 Orchestrator 健康检查

Orchestrator 自身也需要被监控，防止调度脚本挂掉导致整条流水线停止：

```bash
#!/bin/bash
# scripts/orchestrator_watchdog.sh — Orchestrator 看门狗
# 部署为 systemd timer，每 5 分钟检查一次

ORCH_PID_FILE="/var/run/orchestrator.pid"
ORCH_HEALTH="http://localhost:9500/health"
DINGTALK_WEBHOOK="${DINGTALK_WEBHOOK_URL}"

# 检查 Orchestrator 是否存活
if ! curl -sf "$ORCH_HEALTH" > /dev/null 2>&1; then
    echo "$(date) Orchestrator 不可达，尝试重启..."

    # 检查进程是否存在
    if [ -f "$ORCH_PID_FILE" ]; then
        kill -9 $(cat "$ORCH_PID_FILE") 2>/dev/null
    fi

    # 重启
    cd ~/ai-crm
    nohup python -m orchestrator.main > logs/orchestrator.log 2>&1 &
    echo $! > "$ORCH_PID_FILE"

    sleep 10

    # 验证重启成功
    if curl -sf "$ORCH_HEALTH" > /dev/null 2>&1; then
        echo "$(date) Orchestrator 重启成功"
        # 通知钉钉
        curl -s "$DINGTALK_WEBHOOK" \
            -H 'Content-Type: application/json' \
            -d '{"msgtype": "text", "text": {"content": "🟡 Orchestrator 自动重启成功，流水线已恢复运行"}}'
    else
        echo "$(date) Orchestrator 重启失败"
        curl -s "$DINGTALK_WEBHOOK" \
            -H 'Content-Type: application/json' \
            -d '{"msgtype": "text", "text": {"content": "🔴 Orchestrator 重启失败，需要人工排查！\n机器: orchestrator (专用服务器)"}}'
    fi
fi
```

### 7.2 AI Bug 分类器

收到反馈后，先用 LLM 进行智能分类，判断是否为真实 Bug 以及严重程度：

```python
# orchestrator/feedback_triage.py

class AIBugClassifier:
    """使用 LLM 对反馈进行智能分类"""

    TRIAGE_PROMPT = """你是一个 Bug 分类专家。
请分析以下测试反馈，判断:
1. 这是否是一个真实的 Bug？(YES/NO/UNCLEAR)
2. 严重程度: critical / major / minor / suggestion
3. 最可能的根因模块
4. 建议的修复方向

## 任务上下文
- 任务ID: {task_id}
- 任务描述: {task_desc}
- 涉及文件: {files}

## 测试反馈
{description}

请以 JSON 格式回复。"""

    async def triage(self, feedback: TestFeedback) -> dict:
        task = await get_task(feedback.task_id)
        prompt = self.TRIAGE_PROMPT.format(
            task_id=task.task_id,
            task_desc=task.description,
            files=", ".join(task.target_files),
            description=feedback.description
        )
        result = await llm_client.chat(prompt)
        return {
            "is_bug": result["is_bug"],           # YES / NO / UNCLEAR
            "severity": result["severity"],
            "root_module": result["root_module"],
            "fix_direction": result["fix_direction"],
            "confidence": result["confidence"]     # 0.0 ~ 1.0
        }
```

| `NO` | ≥ 0.8 | 自动回复"非 Bug"说明 |

### 7.3 Bug 确认与修复任务自动生成

确认为 Bug 后，FeedbackManager 自动创建修复任务并注入队列：

```python
# orchestrator/feedback_manager.py

class FeedbackManager:
    """管理反馈→修复任务的完整流转"""

    async def process_feedback(self, feedback: TestFeedback):
        """处理新反馈的主流程"""

        # 1. AI 分类
        triage = await self.classifier.triage(feedback)
        feedback.ai_triage_result = triage
        feedback.status = "triaging"
        await self.save(feedback)

        # 2. 根据分类结果决策
        if triage["is_bug"] == "YES" and triage["confidence"] >= 0.7:
            await self._create_fix_task(feedback, triage)
        elif triage["is_bug"] == "NO" and triage["confidence"] >= 0.8:
            await self._auto_reject(feedback, triage)
        else:
            await self._escalate_to_review(feedback, triage)

    async def _create_fix_task(self, feedback, triage):
        """自动生成修复任务"""
        feedback.status = "confirmed"

        # 构建修复任务
        fix_task = CodingTask(
            task_id=f"FIX-{feedback.feedback_id}",
            description=f"修复 Bug: {feedback.description}\n\n"
                       f"AI 分析: {triage['fix_direction']}",
            source="feedback",
            parent_task_id=feedback.task_id,
            priority="high",  # 修复任务默认高优先级
            target_files=triage.get("affected_files", []),
        )
        await task_engine.submit(fix_task)

        feedback.status = "fix_created"
        await self.save(feedback)

        # 通知
        await reporter.send_dingtalk(
            f"🐛 反馈 {feedback.feedback_id} 已确认为 Bug\n"
            f"自动生成修复任务: {fix_task.task_id}\n"
            f"优先级: HIGH | 根因: {triage['root_module']}",
            title="🔧 Bug 修复任务已创建")
```

```python
# orchestrator/ws_broadcast.py

from fastapi import WebSocket
from typing import Set

class ConnectionManager:
    """WebSocket 连接管理器"""

    def __init__(self):
        self.active: Set[WebSocket] = set()

    async def connect(self, ws: WebSocket):
        await ws.accept()
        self.active.add(ws)

    def disconnect(self, ws: WebSocket):
        self.active.discard(ws)

    async def broadcast(self, message: dict):
        """广播消息到所有客户端"""
        dead = []
        for ws in self.active:
            try:
                await ws.send_json(message)
            except Exception:
                dead.append(ws)
        for ws in dead:
            self.active.discard(ws)

manager = ConnectionManager()

# 在任务状态变更时调用
async def notify_task_update(task: CodingTask):
    await manager.broadcast({
        "type": "task_update",
        "task_id": task.task_id,
        "status": task.status.value,
        "machine": task.assigned_machine,
        "progress": task.progress_pct,
        "timestamp": datetime.now().isoformat()
    })
```

```python
class DocChangeTrigger:
    """监听 Git push → 增量解析设计文档 → 生成新任务"""

    async def on_git_push(self, repo_path: str, changed_files: list[str]):
        """Git webhook 回调"""
        doc_files = [f for f in changed_files
                     if f.startswith("docs/") and f.endswith(".md")]

        if not doc_files:
            return

        for doc_file in doc_files:
            # 获取变更的 diff
            diff = await git_ops.get_diff(repo_path, doc_file)

            # 增量解析：只解析变更部分
            new_sections = self.parser.parse_incremental(
                doc_file, diff,
                existing_tasks=await task_engine.get_tasks_by_source(doc_file))

            for section in new_sections:
                task = CodingTask(
                    task_id=self._gen_task_id(section),
                    description=section["description"],
                    source=f"doc_change:{doc_file}",
                    target_files=section.get("target_files", []),
                    acceptance_criteria=section.get("acceptance", ""),
                )
                await task_engine.submit(task)

        await reporter.send_dingtalk(
            f"📝 检测到 {len(doc_files)} 个文档变更\n"
            f"生成 {len(new_sections)} 个新任务",
            title="📋 文档变更触发新任务")
```

### 9.5 触发安全与去重

```python
# orchestrator/task_safety.py

class TaskSafetyGuard:
    """防止重复创建、恶意刷任务"""

    # 频率限制
    RATE_LIMITS = {
        "dingtalk": {"max": 10, "window": 3600},    # 每人每小时最多 10 个
        "api": {"max": 50, "window": 3600},          # API 每小时 50 个
        "doc_change": {"max": 100, "window": 3600},  # 文档变更每小时 100 个
    }

    async def check_before_create(self, task: CodingTask) -> bool:
        """任务创建前检查"""

        # 1. 频率检查
        source_type = task.source.split(":")[0]
        if not await self._check_rate_limit(source_type, task.source):
            raise RateLimitExceeded(f"触发频率超限: {source_type}")

        # 2. 相似任务去重
        similar = await self._find_similar_tasks(task)
        if similar:
            raise DuplicateTask(
                f"发现相似任务: {similar[0].task_id}\n"
                f"描述: {similar[0].description}")

        # 3. 权限检查（钉钉来源验证用户身份）
        if source_type == "dingtalk":
            if not await self._check_permission(task.source):
                raise PermissionDenied("无权限创建任务")

        return True

    async def _find_similar_tasks(self, task: CodingTask) -> list:
        """基于描述的相似度检测（embedding 余弦相似度 > 0.9 视为重复）"""
        embedding = await llm_client.embed(task.description)
        active_tasks = await task_engine.get_active_tasks()
        similar = []
        for t in active_tasks:
            t_emb = await llm_client.embed(t.description)
            if cosine_similarity(embedding, t_emb) > 0.9:
                similar.append(t)
        return similar
```

### 10.2 新增状态定义

在原有 §2.4 的 8 个状态基础上，新增 3 个状态完成闭环：

| 状态 | 含义 | 进入条件 | 退出条件 | 新增? |
|------|------|----------|----------|-------|
| `CREATED` | 任务已创建 | DocParser/API/反馈生成 | 排入队列 | |
| `QUEUED` | 等待分发 | 入队 | 分配到机器 | |
| `DISPATCHED` | 已分发到机器 | SSH 连接成功 | aider 编码完成 | |
| `CODING_DONE` | 编码完成 | aider commit | 进入 Review | |
| `REVIEW` | 三层审查 | 编码完成 | 通过/失败→重试 | |
| `TESTING` | pytest + Judge | Review 通过 | 测试结果 | |
| `JUDGING` | 云端多模型评分 | 测试执行完 | 通过/失败 | |
| `PASSED` | 自动测试通过 | Judge 评分 ≥ 4.0 | 进入发布 | |
| `ESCALATED` | 升级人工处理 | 3次重试仍失败 | 人工处理后关闭 | |
| **`RELEASED`** | **已发布** | **PASSED 后生成 Release Notes** | **进入反馈期** | **✅ 新增** |
| **`FEEDBACK`** | **反馈等待期** | **发布后等待人工测试** | **有反馈→新任务 / 无反馈→关闭** | **✅ 新增** |

```python
from enum import Enum

class TaskStatus(Enum):
    # --- 原有状态 ---
    CREATED    = "created"
    QUEUED     = "queued"
    DISPATCHED = "dispatched"
    CODING_DONE = "coding_done"
    REVIEW     = "review"
    TESTING    = "testing"
    JUDGING    = "judging"
    PASSED     = "passed"
    ESCALATED  = "escalated"
    # --- 新增闭环状态 ---
    RELEASED   = "released"    # 已发布 Release Notes
    FEEDBACK   = "feedback"    # 反馈等待期
    CLOSED     = "closed"      # 彻底完成

# 合法的状态转换
TRANSITIONS = {
    TaskStatus.CREATED:     [TaskStatus.QUEUED],
    TaskStatus.QUEUED:      [TaskStatus.DISPATCHED],
    TaskStatus.DISPATCHED:  [TaskStatus.CODING_DONE],
    TaskStatus.CODING_DONE: [TaskStatus.REVIEW],
    TaskStatus.REVIEW:      [TaskStatus.TESTING, TaskStatus.CODING_DONE],  # 不通过→重编码
    TaskStatus.TESTING:     [TaskStatus.JUDGING],
    TaskStatus.JUDGING:     [TaskStatus.PASSED, TaskStatus.ESCALATED,
                             TaskStatus.CODING_DONE],  # 失败→重编码
    TaskStatus.PASSED:      [TaskStatus.RELEASED],
    TaskStatus.ESCALATED:   [TaskStatus.CLOSED, TaskStatus.QUEUED],  # 人工处理后可重新入队
    TaskStatus.RELEASED:    [TaskStatus.FEEDBACK],
    TaskStatus.FEEDBACK:    [TaskStatus.CLOSED],  # 无Bug→关闭 (有Bug→新建CREATED任务)
    TaskStatus.CLOSED:      [],  # 终态
}

class StateMachine:
    """任务状态机：保证流转合法性 + 触发副作用"""

    def transition(self, task: CodingTask, new_status: TaskStatus):
        """执行状态转换"""
        if new_status not in TRANSITIONS[task.status]:
            raise IllegalTransition(
                f"非法状态转换: {task.status.value} → {new_status.value}")

        old_status = task.status
        task.status = new_status
        task.status_history.append({
            "from": old_status.value,
            "to": new_status.value,
            "timestamp": datetime.now().isoformat()
        })

        # 触发副作用
        asyncio.create_task(self._on_transition(task, old_status, new_status))

    async def _on_transition(self, task, old, new):
        """状态转换时的自动副作用"""

        if new == TaskStatus.RELEASED:
            # PASSED → RELEASED: 自动生成 Release Notes
            await release_note_gen.generate_for_task(task)
            await broadcast({"type": "task_released", "task_id": task.task_id})

        elif new == TaskStatus.FEEDBACK:
            # RELEASED → FEEDBACK: 通知测试人员可验收
            await reporter.send_dingtalk(
                f"✅ 任务 {task.task_id} 已发布，请进行人工验收测试。\n"
                f"如有问题，请 @Bot 反馈 {task.task_id} <问题描述>",
                title="📢 新功能可测试")
            # 启动反馈等待计时器 (默认 24h 无反馈自动 CLOSED)
            asyncio.create_task(self._feedback_timeout(task, hours=24))

        elif new == TaskStatus.CLOSED:
            # 任务彻底完成
            await broadcast({"type": "task_closed", "task_id": task.task_id})

    async def _feedback_timeout(self, task: CodingTask, hours: int):
        """反馈等待超时：无反馈则自动关闭"""
        await asyncio.sleep(hours * 3600)
        if task.status == TaskStatus.FEEDBACK:
            self.transition(task, TaskStatus.CLOSED)
            await reporter.send_dingtalk(
                f"任务 {task.task_id} 反馈期已过 ({hours}h)，无反馈，自动关闭。",
                title="📋 任务自动关闭")
```

### 10.5 与现有模块的集成点

| 现有模块 | 集成变化 |
|----------|----------|
